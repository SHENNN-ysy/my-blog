引言：为什么选择 Composition API

Vue 3 引入的 Composition API（组合式 API）是 Vue 历史上最重要的范式转变之一。它不仅仅是一种新的语法糖，更是一套全新的代码组织哲学。与传统的 Options API 相比，Composition API 解决了三个核心问题：

💡
逻辑复用 — Options API 中通过 Mixins 复用逻辑存在命名冲突和数据来源不清晰的问题；Composition API 通过 Composables（组合式函数）实现透明的状态和逻辑复用。

让我们从最底层开始，逐层拆解 Composition API 的核心机制。

响应式系统核心

Vue 3 的响应式系统建立在 ES6 Proxy 之上。相比 Vue 2 中基于 Object.defineProperty 的实现，Proxy 可以劫持整个对象，监听属性的新增和删除，性能也大幅提升。

一个基本的响应式数据声明如下：

import { ref, reactive, computed, watch } from 'vue'

// ref 用于原始类型
const count = ref(0)
console.log(count.value) // 0

// reactive 用于对象类型
const state = reactive({
  name: 'Vue',
  version: 3
})

// computed — 派生状态
const label = computed(() => `${state.name} v${state.version}`)

// watch — 副作用监听
watch(count, (newVal, oldVal) => {
  console.log(`count changed: ${oldVal} → ${newVal}`)
})
⚠️
解构丢失响应式 — 使用 ES6 解构 const { name } = reactive(state) 会让 name 失去响应式。Vue 3.4+ 提供了 toRef 和 toRefs 来优雅地解决这个问题。

setup 入口与生命周期

setup 函数是 Composition API 的入口点，在组件实例创建之前执行，因此无法访问 this。Vue 3 提供了对应的生命周期钩子：

Options API 钩子 Composition API 钩子 执行时机
beforeCreate — 实例初始化之前
created setup 实例创建后
mounted onMounted DOM 挂载完成
updated onUpdated DOM 更新后
unmounted onUnmounted 组件卸载后

推荐使用 <script setup> 语法糖，它会自动将变量和方法暴露给模板，减少大量样板代码： 

<script setup>
import { ref, onMounted, onUnmounted } from 'vue'

const isVisible = ref(false)

const observer = new IntersectionObserver(...)

onMounted(() => {
  observer.observe(document.querySelector('#target'))
})

onUnmounted(() => {
  observer.disconnect()
})
</script>

ref 与 reactive 的选择

这是 Vue 3 学习者最常纠结的问题。经验法则如下：

  • ref：用于原始类型（string、number、boolean）和需要被重新赋值的对象引用
  • reactive：用于复杂对象，且对象属性会被单独修改的场景
最佳实践：在 TypeScript 项目中，ref 比 reactive 拥有更好的类型推断能力。推荐在新项目中默认使用 ref，在需要声明式状态对象时使用 reactive。

Composables：逻辑复用之道

Composables（组合式函数）是 Vue 3 最强大的特性之一。它将相关逻辑抽离为可复用的函数，类似于 React 中的自定义 Hooks，但借助 Vue 的响应式系统，功能更为强大。

一个标准的 Composable 长这样：

// useWindowSize.ts
import { ref, onMounted, onUnmounted } from 'vue'

export function useWindowSize() {
  const width = ref(window.innerWidth)
  const height = ref(window.innerHeight)

  const handleResize = () => {
    width.value = window.innerWidth
    height.value = window.innerHeight
  }

  onMounted(() => window.addEventListener('resize', handleResize))
  onUnmounted(() => window.removeEventListener('resize', handleResize))

  return { width, height }
}

在组件中使用：

<script setup>
import { useWindowSize } from './composables/useWindowSize'

const { width, height } = useWindowSize()
</script>

<template>
  <div>窗口尺寸: {{ width }} × {{ height }}</div>
</template>
✅
优势：Composables 返回的是响应式引用，使用方无需关心内部实现细节。生命周期自动绑定到当前组件实例，组件卸载时自动清理。

常见的 Composable 场景包括：表单验证（useForm）、数据获取（useFetch）、键盘快捷键（useKeyStroke）、以及主题管理（useTheme）等。

项目结构最佳实践

在大型 Vue 3 项目中，文件组织方式直接影响代码可维护性。推荐以下目录结构：

  • src/composables/ — 所有 Composables，按功能命名（如 useAuth.ts、useArticles.ts）
  • src/components/ — UI 组件，遵循原子设计原则
  • src/utils/ — 纯函数工具，不依赖 Vue 响应式
  • src/types/ — TypeScript 类型定义
命名约定：Composables 必须以 use 开头（如 useCounter），这既是社区共识，也能让 IDE 更容易识别和归类。

对于需要共享的状态，推荐使用 provide/inject 搭配 provide Composable 的方式，避免 Vuex/Pinia 带来的额外复杂度： 

// 全局状态提供者
export function provideUserStore() {
  const user = reactive({ name: '', token: null })

  provide('user', user)

  return {
    user,
    login: async (credentials) => { /* ... */ },
    logout: () => { user.name = '' }
  }
}

常见陷阱与避坑指南

1. 异步 setup 中的响应式丢失

在 async setup() 中，Vue 无法自动追踪所有响应式依赖。确保在异步操作完成后手动触发更新，或使用 watchEffect 来建立依赖追踪。

2. 响应式 proxy 的类型陷阱

reactive() 返回的是一个 Proxy，而非原始对象。如果你需要将响应式对象作为参数传递给普通函数（不感知 Vue 响应式），可以使用 toRaw() 获取原始对象： 

import { reactive, toRaw } from 'vue'

const state = reactive({ count: 0 })

// toRaw 返回原始对象，用于不需要响应式追踪的场景
const plain = toRaw(state)

3. watch 与 watchEffect 的区别

watch 是惰性的，只有当指定依赖变化时才执行回调（类似 componentDidUpdate）；watchEffect 则立即执行，并自动追踪回调中所有响应式依赖（类似 useEffect 的依赖自动收集）。

总结

Vue 3 的 Composition API 是一套经过精心设计的系统。掌握它的关键在于：理解响应式原理、善用 Composables、遵循清晰的目录结构。一旦熟练运用，你会发现代码的可维护性和复用性都有了质的飞跃。

这不是对 Options API 的否定——两者可以共存。但在新建复杂组件时，Composition API 几乎永远是更优选择。
